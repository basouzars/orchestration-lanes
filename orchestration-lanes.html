§1 The problem

Orchestration is not one layer. Different problems ask for different machinery, and the wrong choice imposes either accidental complexity (an engine where a transaction would do) or accidental inconsistency (a method pretending to be a saga). The lanes below are a way to name the choice. Each lane has a distinct owner, duration, persistence model, and visibility profile. Pick by what the work is, not by what is currently nearby in the codebase.

The lanes are not new ideas — they are positions on a spectrum already mapped elsewhere. Lane 3 and Lanes 4/5 descend from the Process Manager pattern in Hohpe and Woolf's Enterprise Integration Patterns (2003). Richardson's saga work distinguishes orchestration-based and choreography-based variants — the former is Lane 3 here; the latter is a sub-mode of Lane 2. "Durable execution," the term Temporal popularised for what workflow engines provide, is what Lane 3 implements by hand and what Lanes 4 and 5 inherit from the engine.

All five lanes share a visual vocabulary: a service is a rectangle, a database is a cylinder, the engine is a hexagonal slab, and event flow is shown in the accent colour. Compensations use the dashed orange line in later diagrams.

§2½ Lanes compose

The lanes are not parallel choices — they layer. A Lane 4 workflow calls journey services that themselves do Lane 1 work and may emit Lane 2 events. A Lane 5 routine composes Lane 1 services. The lanes describe where the orchestration lives, not where every action happens.

Lanes layer. The engine conducts; journey services write; the bus carries side-effects. A real flow touches several lanes at once.
LANE 1

Inline

What it is

One service method, one database transaction, one unit of work. The audit trigger fires on commit; the outbox row commits with the state change. Synchronous HTTP calls to other services live here when they're part of the same logical step. There is no orchestrator — control flow is the call stack.

When to use it

  • The work fits in one bounded context.
  • Latency budget is sub-second.
  • No human acts inside the flow.
  • If a downstream call fails, the whole thing can fail.

When not

  • You'd need compensation if a later step fails.
  • A reviewer or vendor is in the middle of the flow.
  • The transaction holds locks while waiting on a remote system.

What it costs

Cheap in tooling, expensive in coupling if it grows beyond a single context. Lane 1 fails by accretion: a method gradually accumulates remote calls, becomes a distributed transaction in disguise, and the day a downstream times out you discover that what looked inline was always a saga.

Operational, audit, and outbox rows commit together. The relay reads outbox after commit; the rest of the world hears about the change only once it durably exists.

Worked pattern

// Operational write, audit (via trigger), outbox row — atomic.
begin tx
  repo.updateAccount(id, newState)        // trigger writes audit
  outbox.insert("AccountUpdated", payload)
commit tx

// A separate relay polls outbox and publishes.
// If the publish fails, the row stays; the relay retries.
// At-least-once delivery is the contract; consumers must be idempotent.
LANE 2

Event-driven choreography

What it is

Services react to one another's events through a message bus. There is no central coordinator: each service decides what to do when it sees an event, writes its own state, and emits its own events. Every consumer is idempotent because delivery is at-least-once.

When to use it

  • Loosely coupled side-effects: the trigger doesn't need to know who reacts.
  • Reactions are independent and individually retryable.
  • No global "did the whole thing succeed?" question.

When not

  • Compensations distributed across consumers are becoming hard to trace — promote to Lane 3.
  • One consumer is implicitly driving the rest. That's a saga; make it one.
  • You need a single place to ask "where is this process right now?"

What it costs

Cheap to add a new reactor, expensive to reason about end-to-end. Process state is implicit in the union of consumer tables; failure modes are the cross product of partial successes. Tracing exists, but the cost is real and only pays off once the system is genuinely event-shaped.

Sub-mode: choreography-based saga. Lane 2 with compensations distributed across consumers is the choreography-based variant of the saga pattern — same compensation discipline as Lane 3, but with the state machine implicit in the topology of producers and consumers. It works for small flows. When the implicit state machine becomes hard to reason about — typically three or more services with non-trivial compensation paths — promote to Lane 3 and make the state machine explicit.
Producer commits and publishes once; consumers act independently. Idempotency keys on the consumer side absorb the at-least-once contract.

Worked pattern — idempotent consumer

function onAccountUpdated(evt):
  if processed.contains(evt.id): return     // dedupe key

  begin tx
    apply(evt)                              // local state change
    processed.insert(evt.id)              // in same tx
    outbox.insert("DownstreamReacted", …)
  commit tx
LANE 3

Saga as code

What it is

One service owns the process. It holds the state machine in its own database, advances it by sending commands or reacting to events, and runs explicit compensations when a later step fails. The orchestrator is a piece of code that you wrote — not an engine, not a chain of reactions.

When to use it

  • Cross-service work where partial failure must be undone.
  • Duration measured in minutes, occasionally hours.
  • One team owns the process and is comfortable owning the state.

When not

  • The state machine has more branches than a code reader can keep in their head.
  • A human is sitting in the middle with an SLA against them.
  • You need stakeholders outside engineering to see where things are.

What it costs

The compensation logic is yours to write, test, and maintain. So is the recovery story: what happens when the saga crashes mid-flight? The pattern is well understood, but the discipline cost is real — every new step is a new state and a new failure to compensate.

The saga is a state machine in one service's database. Commands fan out; on failure, the compensation path (dashed orange) reverses earlier successful steps.

Worked pattern — state machine with compensation

states: ["reserve", "charge", "confirm", "done", "failed"]

on reserveOk:   state = "charge";  send chargeCmd
on chargeOk:    state = "confirm"; send confirmCmd
on chargeFail:  state = "failed";  compensate(releaseReservation)
on confirmFail: state = "failed";  compensate(refundCharge, releaseReservation)

// Every transition is persisted in the saga's own table before the command is sent.
// On restart the orchestrator reads state and resumes from where it stopped.
LANE 4

Workflow engine — regulatory

What it is

A long-running process running on a workflow engine (n8n in this platform). The execution history is itself a regulatory artifact: examiners can trace, by case, who acted, when, on what input, with what timer or escalation. Human-in-the-loop steps have named SLAs, escalations, and multi-stakeholder visibility.

When to use it

  • Duration is hours to weeks.
  • A human is in the flow with a deadline the regulator cares about.
  • Risk, compliance, and journey owners all need to see the same status.
  • The transcript of the process is what gets shown in an audit.

When not

  • You can keep it inside one service and finish in minutes.
  • Nobody outside engineering will ever ask "where is this case?"

What it costs

An engine in the regulated path adds three concrete costs beyond its own operations. First, in-flight executions are bound to the workflow version they started on; publishing a new version raises the question of migration for running cases. Second, the engine becomes a tier-1 dependency — its outage affects every Lane 4 and Lane 5 workflow simultaneously. Third, the engine and its definitions fall into the SOX and SOC 2 scope of the regulated processes it runs, which expands change-control overhead. Promote to Lane 4 only when the process actually meets the bar.

The engine holds the case state. Human task, timer, and collaborator services are visible nodes in the execution history. State changes still flow through journey services — the engine does not write to systems of record directly.

Worked pattern — workflow definition (conceptual)

workflow "case-review":
  start → call("journey.openCase")
       → humanTask(approver, sla="24h", escalate="supervisor")
       → branch(decision):
           approved → call("journey.markApproved") → end
           rejected → call("journey.markRejected") → end
           timeout  → call("journey.markExpired")  → end
  emits: caseOpened, decisionRecorded, caseClosed
LANE 5

Workflow engine — operational

What it is

Same engine, different governance. Operational and back-office automation owned by ops, compliance, risk, or platform teams. The workflow composes journey APIs and vendor integrations to get a job done — reconciliation, vendor file processing, batch follow-up, periodic compliance routines. State changes to systems of record still pass through journey services. The engine is the conductor, not the writer.

When to use it

  • An operational team needs to compose existing capabilities into a routine.
  • Vendor integrations need retries, schedules, and human-readable runbook semantics.
  • The work is real, recurring, and currently lives in a spreadsheet.

When not

  • You're tempted to write to a system of record directly because it's faster — call the journey service instead.
  • The process is part of a regulatory journey. That's Lane 4.
  • The workflow needs arbitrary code execution (custom script nodes, inline JS) to do its job — these are a separate attack surface even when no system of record is written, and require their own controls before Lane 5 is appropriate.

What it costs

Same engine cost as Lane 4, plus an ownership boundary: ops teams change workflows on a different cadence than engineering. The governance has to say who can modify what, who reviews changes, and how an operational workflow is prevented from quietly becoming part of a regulated path.

Ops owns the workflow. The engine composes journey APIs and vendor APIs. Systems of record remain off-limits to direct writes from the workflow.

Worked pattern — operational routine (conceptual)

workflow "daily-vendor-reconcile":
  schedule: "0 06 * * *"
  steps:
    pull   = vendor.fetchFile(today)
    parsed = parse(pull)
    diffs  = journey.getOpenItems().diff(parsed)
    for d in diffs:
      journey.reconcileItem(d)        // journey writes, not us
    emit summary to ops-channel

§4 Regulatory vs operational

Lane 4 and Lane 5 share the engine. They differ on three questions that decide governance and ownership.

  1. Is the workflow's execution history a regulatory artifact?
  2. Who owns the process — a journey team or an operational team?
  3. Does the workflow need to participate in the journey's audit chain?
When the answer to Q1 is yes, Lane 4. When it's no and ops owns the routine, Lane 5. Edge cases — yes to Q1 but ops-owned, or no to Q1 but examiner-visible — should be escalated to platform + compliance, not resolved by the engineer building the workflow.

A note on BPMN. BPMN engines (Camunda, Zeebe) market the same Lane 4 value proposition — execution history as a regulator-readable artifact — with the additional property that BPMN diagrams are themselves a recognised process specification standard. The lanes here are engine-agnostic; if a process needs the workflow definition to be readable as a regulatory specification (not just the execution history), that is a real reason to evaluate BPMN against the chosen engine. For most Lane 4 work, execution history is sufficient.

Hard rule, both lanes: the engine never writes directly to systems of record. State changes always go through journey services. This is what preserves the journey's audit chain and prevents Lane 5 from quietly becoming a back door around Lane 4.

§5 Common mistakes

Calling any chain of service calls a saga.
The saga pattern specifically solves cross-service consistency under partial failure with explicit compensation. Without compensations you have a service method, not a saga — naming it one hides where the recovery logic should live.
Picking the lane by duration alone.
Duration is a tiebreaker, not a discriminator. The primary signals are audit-visibility, human-in-the-loop, and whether compensations cross service boundaries. A two-hour batch job and a two-hour compliance review live in very different lanes.
Putting everything in the workflow engine because it's there.
Promotion to Lane 4 has real cost — operational, governance, and cognitive. The checklist exists to keep the engine for cases that actually need it. Default to the lightest lane that works.
Treating Lane 5 as a way to bypass journey services.
Both engine lanes share the rule that systems of record are only written through journey services. Reaching past the journey to "just update the row" rebuilds the audit-chain hole the journey exists to close.
Lane 2 choreography that has turned into a hidden orchestrator.
When one service is silently coordinating the rest through events — listening, deciding, and emitting in a way that drives the process — it's a saga. Make it explicit (Lane 3) so the state machine and compensations are visible.
A Lane 1 method that quietly grew sync calls until it became a distributed transaction.
A method that calls three remote services in sequence inside a single transaction is not inline — it's a saga with no compensations and locks held across the network. Either pull the remote work out behind events (Lane 2) or own the state explicitly (Lane 3).
Idempotency on the producer instead of the consumer.
The bus is at-least-once. Consumers carry the dedupe key. A producer that "only emits once" is one redeploy away from emitting twice.
